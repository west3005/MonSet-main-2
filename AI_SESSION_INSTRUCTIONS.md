## 📌 Суть проекта (с первого взгляда)

**MonSet** — промышленное IoT-устройство мониторинга технологических параметров (уровень воды, давление и т.д.) на удалённых объектах.

**Железо:**
- MCU: **STM32F407VET6** (168 MHz, 512 KB Flash, 192 KB RAM), плата STM32F4VE
- Ethernet: **W5500** (SPI1)
- GSM/NB-IoT: **SIM7020C → SIM7020E** (USART2)
- WiFi: **ESP8266** (USART6)
- RTC: **DS3231** (I2C1)
- Датчики: **Modbus RTU** — USART3 RS-485, UART4 RS-485, UART5 RS-232/485
- SD-карта: встроенная в плату, FAT32 via SDIO 1-bit init → 4-bit work
- Спутник: **Iridium SBD NR9602G** (UART5, MAX485 + MAX3232)
- Питание: LiFePo4 24V 15Ah

**Что делает устройство:**
1. Опрашивает датчики по Modbus RTU с конфигурируемой картой регистров (JSON на SD, без перепрошивки)
2. Логирует данные на SD-карту с ротацией по дате
3. Отправляет телеметрию по приоритетной цепочке: **Ethernet W5500 → GSM → WiFi → Iridium SBD**
4. Поддерживает HTTPS, MQTT (ThingsBoard, Home Assistant, Yandex IoT Core, Node-RED), HTTP Webhook
5. При потере связи — буфер в `Backup.json`, отправка после восстановления
6. Веб-интерфейс через ESP8266 / Ethernet / GSM: конфиг, Dashboard, лог, файловый менеджер SD
7. Конфигурация хранится в `runtime_config.json` на SD — применяется без перепрошивки
8. HTTP Basic Auth, TLS, возможность mTLS в будущем
9. Два режима работы: **рабочий** (опрос+отправка) и **веб** (только HTTP, опрос остановлен)
10. WatchDog (IWDG ~32 с) + WatchDog на каждый канал связи

**Ключевые файлы исходников (в репо):**
| Файл | Назначение |
|---|---|
| `Core/Inc/board_pins.hpp` | Источник истины по всем пинам MCU |
| `Core/Inc/config.hpp` | Компиляционные константы |
| `MonSet.ioc` | Конфигурация периферии STM32CubeMX |
| `FATFS/` | FatFS файловая система |
| `Middlewares/Third_Party/` | mbedTLS (TLS 1.2) |
| `Drivers/` | HAL F4 v1.28.0, W5500 ioLibrary |
| `www/` | HTML-страницы веб-интерфейса (хранятся на SD-карте) |

**Структура SD-карты:**
| Файл | Назначение |
|---|---|
| `runtime_config.json` | Все изменяемые параметры устройства |
| `backup.jsn` | Буфер телеметрии при потере связи (JSONL, строки до 240 байт) |
| `log_YYYY-MM-DD.txt` | Ротируемые лог-файлы по дате |
| `www/config.html` | Страница конфигурации |
| `www/index.html` | Главная страница Dashboard |

**Известная проблема SDIO:** CMD25 (multi-block write) несовместим с polling-режимом STM32F4 → TXUNDERR. **Решение:** цикл одиночных CMD24 с `SD_WaitCardReady()` между блоками. `HWFC=0`, `ClockDiv=0`. Подробнее в `README.md`.

## 🔍 Инструкция по анализу кода из репозитория

Перед любой правкой я **обязан** прочитать актуальный файл из ветки `dev`. Порядок действий:

### 1. Получить список файлов проекта

```
GET /repos/west3005/MonSet-main-2/contents/?ref=dev
```
Для директорий (например `Core/Src/`):
```
GET /repos/west3005/MonSet-main-2/contents/Core/Src?ref=dev
```

### 2. Прочитать нужный файл из dev (всегда с актуальным SHA)

При чтении файла через GitHub API я всегда получаю:
- `sha` — хэш текущей версии (нужен для последующего коммита)
- `content` — содержимое файла в base64

Я **никогда не работаю по памяти** — всегда читаю заново перед правкой.

### 3. Ключевые файлы для понимания архитектуры

При старте новой задачи я читаю:
1. `Core/Inc/board_pins.hpp` — все пины, все интерфейсы
2. `Core/Inc/config.hpp` — константы и флаги компиляции
3. Целевой файл `.c`/`.cpp`/`.h` который нужно изменить

### 4. Навигация по структуре репо

```
MonSet-main-2/
├── Core/
│   ├── Inc/          ← Заголовочные файлы (.h, .hpp)
│   └── Src/          ← Исходники (.c, .cpp)
├── Drivers/          ← HAL STM32, W5500 ioLibrary
├── FATFS/            ← FatFS middleware
├── Middlewares/      ← mbedTLS и другие
├── www/              ← HTML веб-интерфейса
├── MonSet.ioc        ← CubeMX конфиг периферии
├── runtime_config.json (на SD, не в репо) ← конфиг устройства
└── README.md         ← Полная документация проекта
```

## 🤝 Правила сессии (запомни)

### Я делаю (человек):
- Коммичу **рабочий код → main**
- В начале сессии говорю ссылку на репо + что нужно исправить/добавить
- Беру изменённый файл из `dev`, тестирую в **STM32CubeIDE**
- Если OK → коммичу в `main`

### Я делаю (AI):
- Читаю актуальный файл из `dev` (**SHA + содержимое** — всегда заново)
- Вношу **точечное изменение** (минимально необходимое)
- Коммичу **только в `dev`** с чётким сообщением (формат: `fix: ...` / `feat: ...` / `refactor: ...`)
- Сообщаю тебе:
  - **какой файл изменён** (путь)
  - **что именно изменено** (кратко и точно)
  - **на что обратить внимание при тестировании**

### ⚠️ Важные правила:
- Я **никогда не коммичу в `main`** — только ты
- Каждое исправление — **отдельный коммит в `dev`**
- Если тест провален — ты описываешь симптом, я **читаю файлы заново** и правлю
- Если задача касается нескольких файлов — уточняю у тебя порядок и приоритет
