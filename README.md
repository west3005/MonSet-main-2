# MonSet

> **Ветка активной разработки** MonSet до продаваемой версии

***

## О проекте

MonSet — промышленное IoT-устройство для мониторинга технологических параметров на объектах.

***

## Аппаратная платформа

| Компонент | Описание |
|---|---|
| **MCU** | STM32F407VET6 на плате STM32F4VE (168 MHz, 512 KB Flash, 192 KB RAM) |
| **Ethernet** | W5500 (SPI1) |
| **GSM / NB-IoT** | SIM7020C → SIM7020E (Cat1-A7670X-V1.02) |
| **WiFi** | ESP8266 (USART6) |
| **RTC** | DS3231 (I2C1) |
| **Датчики** | Modbus RTU: (USART3 RS-485) универсальный, настраивается в конфиге, (UART4 RS-485) универсальный, настраивается в конфиге |
| **Хранение** | Встроенная MicroSD в плату STM32F4VE (FAT32, SDIO 1-bit - инициализация, потом переход на 4-bit - работа) |
| **Спутник** | Iridium SBDNR9602G (UART5 RS-232/485) универсальный, настраивается в конфиге. Аппаратно разведен на 2 линии MAX485 и MAX3232 |

### Схема питания

```
LiFePo4 24V 15Ah
    │
    ├─ 24V ──────────────── Датчики (415 погружной, ULM31 радарный)
    │
    ├─ DC-DC 24V → 5V ───── ESP8266, W5500, DS3231, GSM-модем
    │                       (далее LDO 5V → 3.3V для каждого модуля)
    │
    └─ DC-DC 5V → 3.3V ──── STM32F407VET6
```

## Функционал устройства

### Сбор данных

1. Считывает данные с датчиков по Modbus RTU (UART/RS-485)
2. Синхронизирует время через NTP или DS3231
3. Логирует данные на SD-карту с ротацией по дате
4. Усреднение за выбранное количество измерений
5. Все опросы датчиков сохраняются на SD в файл `Backup.json`
6. В конфиге — настройка нулевой отметки (уровень моря), коэффициентов умножения/деления для конвертации

### Передача данных

7. Отправляет телеметрию на ThingsBoard через GSM и Ethernet W5500 (HTTPS и MQTT — выбор через конфиг)
8. Поддержка дополнительных MQTT-брокеров: Home Assistant (Mosquitto), Node-RED, Yandex IoT Core, свой сервер
9. HTTP Webhook — отправка на произвольный URL (Zapier, IFTTT, собственный сервер) по событию или расписанию
10. **Приоритетная цепочка каналов** Данный режим включается/отключается и настраивается в конфиге:
    - Ethernet W5500 → GSM → WiFi ESP8266 → Iridium SBD
    - Автопереключение при недоступности, возврат на приоритетный при восстановлении
11. WatchDog на каждый канал связи — GSM, Ethernet, WiFi, IRIDIUM
12. При неудачной отправке данные сохраняются в `Backup.json` и отправляются после восстановления связи
13. После удачной отправки буферные данные стираются
14. Отдельная дискретность отправки `Backup.json` на сервер (независима от дискретности опроса)

### Конфигурация

15. Конфиг хранится на SD-карте (`runtime_config.json`)
16. Удалённое конфигурирование через WiFi ESP8266, по Ethernet и по GSM
17. Конфигурируемая карта регистров Modbus в JSON на SD: адрес, функция, тип данных (`int16`/`uint32`/`float`), масштабирующий коэффициент, единица измерения — без перепрошивки
18. Настройка каждого порта отдельно: параметры порта, параметры Modbus, несколько датчиков с разными адресами на одном канале
19. Поддержка Modbus TCP через W5500 — опрос устройств в локальной сети
20. Сам может выступать в качестве Modbus TCP slave и передавать измерения

### Веб-интерфейс

21. Веб-интерфейс через ESP8266, GSM и Ethernet (W5500) — с Runtime-логами и конфигом
22. Все страницы веб-интерфейса лежат в папке `www/` в корне SD карты
23. Все параметры устройства настраиваются через веб-интерфейс
24. Адаптивный интерфейс — корректно открывается на телефоне и ПК
25. Настройки разбиты по категориям в отдельных окнах/вкладках: Основные, Каналы, Modbus, Протокол, Расписание, Система
26. Отображение последних значений датчиков без необходимости ThingsBoard
27. Ведение лога на отдельной вкладке
28. Скачивание логов через веб-интерфейс
29. Экспорт данных из `Backup.json` в `.json` и CSV (для Excel / Google Sheets)
30. Индикация уровня заряда батареи через ADC с отправкой на сервер и предупреждением при низком заряде
31. Включение и отключение каналов измерения
32. Безопасное удалённое управление с логином и паролем
33. **Мастер первоначальной настройки** (Captive Portal через ESP8266):
    - *Простая версия* — подключение к WiFi, APN, MQTT-брокер, токен ThingsBoard, сервер
    - *Расширенная версия для разработчика* — все возможные настройки

### Безопасность

34. TLS + веб-интерфейс с авторизацией (HTTP Basic Auth)
35. Логин и пароль хранятся в `runtime_config.json` на SD (`web_user`, `web_pass`)
36. Смена пароля через вкладку настроек — данные записываются на SD немедленно
37. *(Опция, на будущее)* Mutual TLS (mTLS) — двусторонняя аутентификация, хранение ключей во внутренней Flash STM32

### Надёжность

38. WatchDog для всех функций программы и автовосстановления (IWDG ~32 с)
39. Поддержка Modbus TCP Server — устройство как Modbus-slave для SCADA (GE iFIX, WinCC, InTouch)

## Логика работы веб-сервера

### Режим работы

Устройство работает в двух режимах — **рабочий** и **веб**. По умолчанию оно опрашивает датчики, пишет на SD и отправляет данные. Как только браузер открывает страницу — устройство переходит в веб-режим.

**Рабочий режим** (нормальный):
- Каждые N секунд: опрос Modbus → запись на SD → отправка на сервер
- Веб-сервер тикает каждые 5 мс в фоне — ждёт подключения

**Веб-режим** (как только браузер подключился):
- Рабочий poll немедленно останавливается
- Все ресурсы отдаются веб-серверу — отвечает на запросы браузера без задержек
- Modbus, отправка данных, backup — не выполняются пока веб активен
- После `web_idle_timeout_s` секунд бездействия браузера — устройство автоматически возвращается в рабочий режим

### Конфиг (хранение и обмен)

- **Хранится только на SD карте**: `0:/www/runtime_config.json`
- **При старте устройства**: один раз читается с SD в RAM (`loadFromSd`)
- **При каждом открытии страницы**: браузер делает `GET /api/config` → устройство отдаёт конфиг **из RAM** (он всегда актуален)
- **При нажатии «Сохранить»**: браузер отправляет `POST /api/config` → данные записываются в RAM и сразу на SD → страница перечитывает конфиг и показывает актуальные значения
- **SD — единственное постоянное хранилище**: если файл не найден на SD — страница отдаст ошибку 503

### Схема одним взглядом

```
Старт
  └─ loadFromSd() → RAM  (один раз)

Браузер открыл страницу
  └─ GET /www/config.html ← читается файл с SD
  └─ GET /api/config      ← отдаётся из RAM
  └─ poll остановлен, веб работает

Пользователь нажал Сохранить
  └─ POST /api/config (JSON)
       ├─ RAM обновлён
       └─ SD обновлён (0:/www/runtime_config.json)
  └─ GET /api/config      ← страница перечитывает из RAM

Браузер закрыт / timeout
  └─ веб-режим выключен → poll возобновляется
```

### Авторизация и смена пароля

- Вход защищён логином и паролем (HTTP Basic Auth)
- Логин и пароль хранятся в `runtime_config.json` на SD (`web_user`, `web_pass`)
- Смена пароля — через вкладку настроек веб-интерфейса: ввести новый пароль → нажать Сохранить → данные записываются на SD → при следующем входе используется новый пароль
- Если авторизация отключена (`web_auth_enabled = false`) — страница открывается без пароля
- При неверном пароле сервер отвечает `401 Unauthorized` и браузер показывает диалог повторного входа

### Вкладка Dashboard

- При переходе на вкладку браузер делает `GET /api/sensors` — устройство отдаёт **текущие рантайм-значения** прямо из RAM: последнее измерение датчика, заряд батареи, статус каналов
- График последних 10 минут: данные хранятся в кольцевом буфере в RAM (или читаются из backup-файла на SD), браузер запрашивает их через `GET /api/sensors?history=1`
- Обновление в реальном времени — браузер делает polling каждые N секунд пока открыта вкладка
- Отображает: значение датчика, временну́ю метку, уровень заряда батареи, статус отправки (OK / backup / ошибка), статус каналов (ETH / GSM / WiFi / Iridium)
- Кнопка **«Тест — отправить текущее значение на сервер»**: принудительно инициирует одиночную отправку текущего показания через активный канал, результат (OK / ошибка) отображается на странице

### Вкладка Лог

- Полноценный лог как в UART — все строки `[INFO]`, `[WARN]`, `[ERROR]`, `[DATA]`
- Браузер запрашивает `GET /api/logs` — устройство отдаёт содержимое кольцевого лог-буфера из RAM (последние N строк)
- Автообновление каждые 2 секунды пока вкладка открыта
- Кнопка **Скачать лог** — `GET /api/logs/export` — отдаёт полный лог файлом `.txt`
- Кнопка **Очистить** — `POST /api/logs/clear`

### Вкладка Файлы (SD-карта)

- Файловый менеджер SD-карты прямо в браузере
- `GET /api/files?path=/` — листинг директории: папки и файлы с размерами
- Навигация по папкам — клик по папке открывает её содержимое
- **Скачивание** любого файла — `GET /api/download?path=/имя_файла`
- **Загрузка** файлов на SD — `POST /api/upload?path=/папка` с файлом в теле запроса; позволяет обновить `config.html`, `index.html` и другие файлы прямо из браузера без извлечения карты
- Отображает: `runtime_config.json`, `backup.jsonl`, лог-файлы, HTML-страницы в `www/`
- Позволяет забрать backup-данные для экспорта в Excel / Google Sheets без извлечения карты

### Мастер первоначальной настройки (Captive Portal через ESP8266)

- При первом включении (или если `runtime_config.json` не найден на SD) ESP8266 поднимает точку доступа
- Подключившись к AP, браузер автоматически открывает страницу настройки (Captive Portal)
- **Один режим — расширенный**: читает все параметры из `runtime_config.json` на SD, отображает полную форму со всеми настройками — IP, каналы, APN, WiFi, MQTT-брокер, токен ThingsBoard, Modbus, расписание, безопасность
- После заполнения и нажатия «Сохранить» — данные записываются обратно в `runtime_config.json` на SD, устройство перезагружается и стартует с новыми настройками
- Если файл конфига уже есть — форма заполняется его текущими значениями (не дефолтами)

### SDIO — MicroSD карта (встроена в плату STM32F4VE)

## SDIO Write — Известная проблема и решение

### Симптом

```
[ERROR] SDIO: TXUNDERR ...
CardState=6 (PRG — карта занята записью)
```

### Причина

Архитектурная несовместимость CMD25 (multi-block write) с polling-режимом STM32F4:
карта уходит в состояние `PRG`, DPSM не останавливается, FIFO пустеет → `TXUNDERR`.

### Решение

Цикл одиночных CMD24 с `SD_WaitCardReady()` между блоками вместо CMD25:

```c
for (uint32_t i = 0; i < NumberOfBlocks; i++) {
    err = SD_WaitCardReady(SD_TIMEOUT);
    if (err != HAL_OK) return err;
    err = HAL_SD_WriteBlocks(&hsd, pData + i*512, BlockAdd + i, 1, SD_TIMEOUT);
    if (err != HAL_OK) return err;
}
```

### Дополнительные настройки

| Параметр | Значение | Причина |
|---|---|---|
| `HWFC` | 0 (выключен) | Hardware Flow Control вызывает ложный TXUNDERR |
| `ClockDiv` | 0 | максимальная скорость, без делителя |
| `DCTRL` | сброс перед записью | очистка состояния DPSM |

### Диагностика

Если проблема вернётся — в логе будет:
```
[HAL_SD] DPSM ON: STA=0x00045000
[HAL_SD] while#1: TXFIFOHE=1 dataremaining=512
```
`TXFIFOHE=1` при `dataremaining=512` означает что FIFO пуст до начала записи — признак TXUNDERR.

### Хронология патчей

- `v8` — первая диагностика, выявлен TXUNDERR
- `v12-diag` — расширенное логирование DPSM/STA
- `v13` — финальный фикс: CMD24 loop + WaitCardReady

## Требования и конфигурация

### runtime_config.json на SD-карте

Все изменяемые параметры хранятся в `runtime_config.json` на SD-карте и применяются без перепрошивки:

- APN, логин/пароль оператора
- URL сервера, токен авторизации, протокол (HTTPS / MQTT)
- Дискретность опроса датчиков и дискретность отправки
- Приоритеты каналов связи
- Карта регистров Modbus для каждого порта
- Нулевая отметка и коэффициенты масштабирования
- Настройки веб-доступа: `web_user`, `web_pass`, `web_auth_enabled`, `ntp_enabled`

## Выбор GSM-модуля

> Текущий модуль: **SIM7020C**

### 🥇 SIM7020E — оптимальный выбор

Прямая pin-to-pin и AT-команды совместимая замена SIM7020C.
Добавляет диапазоны **B20** и **B28** — полное покрытие МТС, МегаФон, Билайн, Теле2 в России.
Код драйвера MonSet переносится без изменений: те же `AT+CMQTTCONNECT`, `AT+CMQTTPUB`.
Потребление в PSM ~1 µA — идеально для батарейного питания.

### 🥈 Quectel BC660K-GL — лучший NB-IoT чип

NB-IoT Cat NB2 (3GPP Rel.14).
Поддерживает B1/B2/B3/B5/B8/B12/B13/B14/B17/B18/B19/**B20**/B25/B28/B66/B70/B85 — самый широкий охват.
Встроенный eSIM, потребление в активном режиме ~5 mA.
Требует нового драйвера под Quectel AT-команды.

### 🥉 Quectel BG95-M3 — лучший для отдалённых объектов

Единственный с **2G EGPRS fallback** — при отсутствии NB-IoT автоматически переключается на GPRS.
Поддержка Cat-M1 даёт скорость до 588 kbps для FOTA-обновлений.
Продаётся в России через Aurora Evernet.
Потребление ~18 mA в активном режиме — оправдано для объектов без NB-IoT покрытия.
